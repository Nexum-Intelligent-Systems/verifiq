## What was built

The council pipeline that wires the Phase 2 agents into one resumable workflow,
per `verifiq-prompts/03_review_workflow.md` and `20_platform_architecture.md` §2,
plus the two council engines the pipeline needs that didn't yet exist.

### Council engines (Stages 5–6)

- **`src/agents/challenge.ts` — Peer Challenge agent (Stage 5).** Loads master +
  `07_council_prompts.md` §07.1, calls the `peer-challenge` LLM role (a different
  model family — file 02), and returns one `ChallengeRecord` per finding
  (decision / reason / revised risk / interface / action). Parsing is defensive:
  a malformed reply degrades to "no challenge", never breaks the run.
- **`src/agents/adjudicate.ts` — Adjudicator agent (Stage 6).** Produces the
  accepted register by **code-deterministic** file-06 rules: delete
  ownerless / no-consequence / out-of-scope / unsourced findings; merge
  duplicates; apply peer escalations/downgrades; triple-jeopardy
  (fire + accessibility + life-safety) → Critical. Records pre/post state on
  every decision (file 06 audit trail). The `adjudicator` LLM role is consulted
  best-effort for the model stamp only — it never drives deletion or the rating.

### Orchestrator (`src/orchestrator/`)

- **`queue.ts` — `InMemoryJobQueue`.** Mirrors the `jobs` table model (file 20
  §2): idempotent enqueue, `depends_on` ordering, retry-with-backoff (2s/4s/8s),
  and **per-tree isolation** — a job whose dependency permanently failed is
  marked `blocked` while independent trees still complete. Uses a virtual clock
  so backoff is deterministic and instant in tests.
- **`workflow.ts` — `Orchestrator`.** Runs review (per discipline) → peer
  challenge → adjudicate → chair report as a job DAG.
  - **Per-discipline isolation:** a review that fails after the agent's own
    provider failover is recorded (`discipline_review_failed`) and the pack
    proceeds with the survivors — it does not fail the scan (file 20).
  - **Resumable:** every stage persists output, advances `scan_state`
    (file 20 §5), and is marked complete; a second `run()` reloads state and
    skips finished stages (idempotent — no agent re-invocation).
  - **Decoupled:** all reads/writes go through the injected `PersistencePort`;
    the LLM goes through the agents' injected client. No Convex import.
- **`types.ts` / `memory.ts` / `index.ts`** — the stage model, the
  `PersistencePort` interface, an in-memory reference implementation, and the
  barrel.

### Convex glue (`src/convex/jobs.ts`)

Minimal persistent job-queue functions mirroring file 20 §2: `enqueueJob`
(idempotent on `idempotency_key`), `claimNextRunnable` (dependency-gated),
`completeJob`, `failJob` (retry backoff → `failed` after max attempts),
`advanceScanState`, `listJobs`. The scheduled `tick` that drives these is Phase 4.

## Deviations / decisions

1. **Adjudication is code-deterministic, LLM-assisted.** Like the Chair's
   rating (Phase 2), the register is decided in code so the file-06 invariants
   hold regardless of model variance; the adjudicator model is still exercised
   (role stamp + future rationale). Rationale text is code-generated by default.
2. **Orchestrator resilience over hard DAG failure.** A permanently-failed
   discipline review does not block the downstream stages — the pack reports
   from the survivors and logs the failure. The queue's strict
   dependency-blocking behaviour is still implemented and tested separately.
3. **Classify stage not wired here.** Phase 3 assumes documents arrive grouped
   by discipline (`documentsByDiscipline`). The title-block classifier
   (file 20 §3) is Phase 4 alongside the scheduled `tick`.
4. **In-memory queue + persistence are the tested core.** The Convex-backed
   `PersistencePort` and the `tick` scheduler that runs `jobs.ts` are Phase 4
   (they need a deployment to exercise).
